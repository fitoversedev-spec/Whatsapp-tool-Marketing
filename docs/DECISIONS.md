## 2026-07-20 — Performance audit: CRM analytics route + bulk import dedupe

Profile-first pass over the new CRM code against the stated 100 reps × 5000 customers target. Current real data is tiny (35 deals, 21 accounts, 270 line items), so most of this is query-plan/index reasoning rather than a measured wall-clock win — called out per-item below. The `crm-dev` Neon branch in `.env` is unreachable (auth failure against its pooler host — looks expired/deleted, not a credentials typo, since the same script connects fine against the production URL). All live checks below are therefore read-only Prisma-query-logged calls against production's existing rows (zero writes for measurement; the one write-path check — intra-batch dedupe in `commitRow()` — was verified by code review only, not a live write test).

**`/api/crm/analytics` ran 4 of its 9 analytics functions for nothing.** `CrmAnalyticsClient.tsx`'s 4 tabs only ever read `salesActivity`, `funnel`, `products`, `sources`, `stageVelocity` off the response — `geography`, `customerSegments`, `timelineMetrics`, and `forecast` were computed and serialized into the JSON on every load and every date-range change but never referenced anywhere in the client (confirmed by grep — zero occurrences of `.geography`/`.customers`/`.timelines`/`.forecast` in that file). Team Performance's own route still calls all 9 — its client (`TeamAnalyticsClient.tsx`) actually renders those 4 tabs — so they stay there unchanged. Removed the 4 dead calls from `src/app/api/crm/analytics/route.ts` only.

**`productAnalytics()` and `timelineMetrics()` had no date bound in SQL at all.** Both read "fetch broad, filter in JS" like the other analytics functions, but unlike `sources.ts`/`customers.ts`/`geography.ts` (which all filter by `createdAt`/`closedAt` range at the query level), these two had *no* `where` date filter whatsoever — `productAnalytics()`'s `DealLineItem.findMany` and `timelineMetrics()`'s main `Deal.findMany` re-fetched every historical row on every call regardless of the selected range, then filtered in JS. Bounded both with an `OR` across the specific timestamp fields each function actually reads downstream (enquiryAt/quotation.sentAt/closedAt for products; enquiryAt/closedAt for timelines) — a row matching none of those branches was already contributing nothing to the output, so this is a pure narrowing. Verified two ways on live data: (1) full JSON output diffed byte-for-byte identical before/after for the same wide range (timelines differed only in the wall-clock-dependent `daysSinceChange` field, expected since `now()` moves between runs); (2) a range with no matching data now fetches 0 rows where it previously always fetched the whole table (270 line items / 35 deals). `timelineMetrics()`'s separate `openDeals` query for "stuck deals" is correctly left unbounded — that one is inherently about currently-open deals, not the date range.

**`DealStageHistory` was missing an index for `stageVelocity()`.** Confirmed via both `schema.prisma` and a live `pg_indexes` catalog query against production: the table has `[dealId, changedAt]` and `[toStageId, changedAt]` but nothing on `fromStageId`, which is exactly what `stageVelocity()` filters + groups by. Added `@@index([fromStageId, changedAt])`, mirroring the existing `toStageId` sibling index. Append-only table (12 rows today, grows with every stage transition forever) — not yet a measured slowdown, but a real, growing full-scan risk on a new query pattern. Needs `npx prisma db push` to actually take effect; not pushed as part of this pass.

**`CourtImage.dealId` was missing an index too**, found while checking `getRepDeals()` (below) for N+1 risk — its `courtImages` include filters `court_images` by `deal_id IN (...)`. The model's own comment says `dealId` is "same pattern as `Quotation.dealId`," but `Quotation` has `[dealId, version]` and `CourtImage` had nothing. Added `@@index([dealId])`. Same "needs `db push`" caveat as above.

**`getRepDeals()` is *not* an N+1**, checked directly rather than assumed. Ran it with `PRISMA_LOG_QUERIES=1` against the real top-owning rep (22 deals): every relation, including the two `take: 1` relations (`activities`, `reminders`), came back as exactly one batched `WHERE deal_id IN (...)` query — Prisma correctly folds `take`-limited to-many includes into a single query here, it does not issue one query per parent row for this shape. Query count is flat regardless of deal count; no fix made. (Real, smaller nuance found in passing: those two `take: 1` relations fetch all matching rows per included deal and trim to 1 client-side rather than pushing `LIMIT 1` into SQL per group — normal, documented Prisma behavior without the `relationJoins` preview feature, bounded by a rep's total historical activity/reminder count, not worth the bigger change of turning on a preview feature for.)

**Bulk import (`/api/import/preview` and `/api/import/commit`) had a real O(n²)-shaped bug**, the most severe thing found this pass. Both routes loop over every uploaded row (up to 2000) calling `validateRow()`, and for the CONTACTS and LEADS targets that function re-ran an *unbounded* `findMany` — the entire `account_contacts` table (CONTACTS, phone dedupe) or the entire `leads` table (LEADS, no filter at all) — on every single row, because phone numbers aren't stored normalized so an exact SQL match can't be trusted across formats. Measured directly (query-logged, read-only, against production): 5 simulated rows produced 5 full-table fetches of each; confirmed the fetched query had no `LIMIT`/narrowing `WHERE` beyond phone-not-null. At the target scale this compounds with both the import size and the existing table size on every row. Fixed by building the normalized-phone lookup **once** per import call (`buildDedupeContext()` in `src/lib/import/dedupe.ts`) instead of once per row; `commitRow()` registers newly-created rows into the same map so same-batch duplicates (two rows in one file sharing a phone number) are still caught, matching the original re-fetch-every-time behavior. Re-measured: 5 rows now produce exactly 1 full-table fetch. `findAccountContactDuplicate()` (`src/lib/crm/accounts.ts`) took an *optional* second parameter for this — its existing single-row caller (the New Contact form) is untouched. Note: the LEADS import target is currently unreachable from the UI picker (see "General CRM Leads section removed" below) — this fix's live impact today is the CONTACTS path; the LEADS half stays correct for whenever that target is re-enabled, consistent with that entry's "one-line change, not a rebuild" intent.

**Checked and left alone, no measured or reasoned issue found:** `sources.ts`, `customers.ts`, `geography.ts` all already bound their queries by the selected date range in SQL before aggregating in JS — the "fetch a bounded set, reduce with Maps" pattern itself is fine at this scale, only the two *unbounded* fetches above were the actual problem. `src/lib/crm/timeline.ts` (`getUnifiedTimeline`) is already `take`-limited. The Prisma client is already a proper module-level singleton on the pooled (`-pooler`) Neon URL with `pgbouncer=true`/timeouts set (fixed in an earlier session, unchanged here).

**Noted but not changed (no clear enough justification yet for a schema change):** `timelineMetrics()`'s separate `historyRows` query against `deal_stage_history` (changedAt range + `durationInFromStageSeconds not null`, no `fromStageId`/`toStageId` filter) doesn't fully match either existing index or the new one added above; `resolveAccountByName`/`resolveAccountId`'s per-row `Account.findFirst` by case-insensitive name during CONTACTS/DEALS import has no supporting index (a case-insensitive/`ILIKE` match would need a functional or trigram index, a bigger change) and the accounts table's real growth rate at target scale is unclear. Both are small enough today (12 and 21 rows respectively) that adding more indexes on inferred-only risk felt like guessing rather than fixing a confirmed bottleneck — worth revisiting once there's real data volume to check against.

## 2026-07-20 — General CRM Leads section removed (reversibly)

Removed after an explicit before/after confirmation with the user. Evidence: all 7 existing `Lead` rows were identifiably test data (three literally named `"WhatsApp lead"` — inside a section whose entire purpose was to exclude WhatsApp leads — plus repeated names/phone numbers matching other test data from this session, all sharing one lead source, all created in the same second). Zero had ever been converted to a deal, in over a week of the CRM being live.

**Reversible by design** — the `Lead` Prisma model/table was NOT dropped, only every user-facing surface:
- Deleted `/crm/leads` page + client, and the now-orphaned `/api/leads` (general CRUD) and `/api/leads/[id]/convert` routes. **Did not touch** `/api/leads/[id]/route.ts` — despite the shared URL prefix that one operates on the unrelated `BotLead` model and is still needed by the Sidebar's own "Bot leads" page.
- Removed the "Leads" nav tab, All Tools tile, and its global-search integration (API query + UI section).
- Removed `LEADS` from the import wizard's selectable targets — `mapping.ts`/`dedupe.ts` keep the underlying field-mapping and commit logic intact, just unreachable from the picker, so re-adding it later is a one-line change, not a rebuild. Bulk-importing a raw prospect list now goes through the Contacts target instead, which already captures everything Leads did (plus Location/Customer type/Business type/Lead source/Notes, added earlier this session).
- **Found and fixed a real dependency in the process**: the WhatsApp bot's `new lead <name> <city> <phone>` staff command (§10) wrote directly to `prisma.lead.create()`. Left as-is, it would have kept accepting the command, replying "✅ Lead created", and silently writing into a table with no page to view it from — a real regression, not a cosmetic one. Repointed to create a real Account + AccountContact instead (same auto-named-account pattern the simplified New Contact form already uses), so the command still lands somewhere visible. Command syntax and reply format are otherwise unchanged from the rep's point of view.
- Deleted the 7 confirmed-junk rows — the delete script refused to run if any row had a `convertedDealId` set, as a hard safety check; none did.

Verified with 9 direct assertions on the bot-command repoint specifically (calling `executeStaffCommand()` directly, per its own header comment — no Meta API involved): a real Account + primary AccountContact created with the right fields, the reply text updated, and confirmed zero Lead rows get created going forward. Typecheck clean; re-ran `scripts/rbac-matrix-check.ts` after editing its search-coverage row description (8 pass / 0 fail / 8 not-implemented / 2 divergences, unchanged from before — the edit was descriptive only, not a logic change).

## 2026-07-20 — Contact Detail: Log Activity, inline stage change, New Deal — and a real sync bug fixed along the way

Contact Detail had no way to log an activity, change a deal's stage, or create a bare deal — only read-only lists plus the existing quote/design/product-interest quick-actions. Added all three:

- **New Deal** quick-action — the existing "no deal yet" prompt (previously only reachable as a side-effect of the quote/design quick-actions) is now also its own direct button.
- **Log Activity** — new modal, optionally tied to one of the contact's deals. New `POST /api/crm/activities` handles this (the existing `/api/deals/[id]/activities` is deal-scoped and can't log against a contact with no deal yet); it verifies a passed `dealId` actually belongs to the same account before linking it, rather than trusting the client.
- **Inline stage change** — the Deals card's stage badge is now a real `<select>`, going through the same `POST /api/deals/[id]/stage` → `transitionDeal()` path the Deals list already uses (won/loss-reason stages prompt the same closeout details first). Since `transitionDeal()` is the sole place allowed to write `Deal.currentStageId`, this is a real answer to "does it sync everywhere" — yes, because there's exactly one Deal row and one mutation function behind every surface (Deal Detail, Deals list, Pipeline, Contact Detail, analytics); nothing is a separate copy that could drift. Worth knowing: Deal Detail itself doesn't have this same inline control today (only the Deals list did, until now) — a real, pre-existing gap, not something this fix needed to touch.

**Found while wiring Log Activity — a genuine, real sync bug:** `POST /api/deals/[id]/activities` (the existing Deal Detail "Log activity" modal) never set `accountContactId` or `accountId` on the Activity it created, even though the deal it belongs to has both readily available. Practically: logging an activity from a Deal never showed up on that deal's own primary contact's Contact Detail page, or on its Company Detail page — both read activities by `accountContactId`/`accountId`, and both were always empty for anything logged this way. Fixed by deriving both from the deal at creation time. Verified with 6 direct assertions, including proving the fix makes a deal-logged activity actually appear in the Contact page's own query, and that the new contact-anchored endpoint correctly rejects a `dealId` from an unrelated account rather than trusting it — disposable rows, cleaned up after.

## 2026-07-20 — Contacts bulk-import template aligned with the New Contact form

The downloadable `contacts-import-template.xlsx` and the wizard's mapping/commit step only ever captured 5 columns (Company name, Contact name, Phone, Email, Designation) — untouched since Phase 2, predating everything the New Contact form has grown since (Location, Customer type, Business type, Lead source, Notes). Added all 5 as new `IMPORT_FIELDS.CONTACTS` entries; the wizard's mapping UI and template download are both already fully data-driven off that config, so no separate UI change was needed.

**Lead stage / auto-deal-creation deliberately left out.** The manual form's "pick a stage → a Deal gets created alongside the contact" only works because it's a single record. Doing the same during a spreadsheet import would mean one Contacts-import batch silently also creating Deal rows — breaking the wizard's own established rule that a batch is always single-target (Phase 2 decision). Someone bulk-creating deals already has the separate DEALS import target for that.

**Lead source has no dedicated column to write to.** Only `Deal.leadSourceId` exists — `AccountContact` has none — and since this import path deliberately never creates a Deal, there's nowhere to put a real FK. Composed it into the contact's `notes` field instead (`"Lead source: Meta Ads — Instagram"`), the exact same fallback the manual form already uses for its own "source detail" field — not a new pattern, just extended to cover this case too. Customer type gets the same fallback treatment when the typed text doesn't match any real taxonomy entry (`"Customer type (unmatched): ..."`) rather than silently discarding it.

**Company reuse now applies classification even on an existing account**, not just on create — matches `resolveAccountId`'s same "a later, more-informed value wins" rule already used by the WhatsApp quote flow, reimplemented locally in the import wizard's own `resolveAccountByName` rather than importing that other module's helper (its side effect of auto-creating a primary contact when a phone is given would have double-created a contact alongside this path's own explicit `accountContact.create`).

Verified with 13 direct assertions: template header count/labels, a new-company row correctly setting city/customerProfileId/businessType (including lowercase `"b2b"` normalizing to `"B2B"`), reusing an existing company by name updating its classification to a later row's value, and an unmatched customer-type string landing in notes instead of vanishing — disposable rows, cleaned up after.

## 2026-07-20 — Stage velocity, flooring-only product tracking, per-rep drill-down, contact reassignment

**Stage velocity (overall + per rep).** New `stageVelocity()` in `timelines.ts`, reusing the file's existing `stat()`/`DurationStat` helpers. `DealStageHistory.durationInFromStageSeconds` already recorded exactly this on every real transition (written by `transitionDeal.ts`) but was only ever surfaced as one blended average across all stages combined (`timelineMetrics().timeInStage`) — never rendered in the CRM Analytics UI at all despite the API already returning it. New function groups by `fromStageId` instead, giving a real per-stage breakdown ("how long to move from Enquiry Received to the next stage", etc.), ordered by `FunnelStage.sortOrder`. Shown on Overall performance (unscoped) and on the new per-rep page (scoped to that one owner) — same function, different `ownerIds`.

**Product tracking scoped to flooring, with a city breakdown.** Confirmed: yes, tracked via `Deal.siteCity` — `productAnalytics()` already computed a `cityHeatmap` (`{productName, city, wonValue, enquiries}`) but it was silently dropped by the UI, never rendered (the exact kind of deferred gap flagged at the end of Phase 5). Surfaced it as a new table under Best-selling products.

For "flooring only": `Product.type` alone is unreliable in both directions — checked real data first rather than assuming. 252 of 270 real line items have no matched `productId` at all (rate-sheet category labels like "Artificial Turf for Multisports", not catalogue picks), and 2 real catalogue products are flat-out mistyped as `type: "flooring"` despite being a lighting system and a netting product. Classification instead matches the vocabulary `Product.type`'s own schema comment already defines as flooring (turf / acrylic / PPE tile / PVC) against whichever text is available — the line's label, or its matched product's name. Verified against real data: Fencing/Sub Base/Nylon Net/Lightings correctly excluded, every remaining product name is genuinely turf/acrylic/flooring.

**Per-rep drill-down page.** Clicking a rep's name in Individual performance now opens `/crm/analytics/rep/[userId]` (admin-only, same gate as the rest of CRM Analytics): that rep's stage-velocity breakdown, plus a 4-column roster — customer (linked to the deal), quote/court-design/product-interest (each a real link: quotation → its PDF route, design → its stored image, products → the interested-product list), current stage, and latest activity note + next upcoming reminder. XLSX export via the same `downloadXlsx` helper already used elsewhere. This is a full roster (every deal the rep owns), not date-windowed — the date range only scopes the stage-velocity card on the same page.

**CRM Contacts ownership — confirmed already correct, reassignment was the real gap.** Checked before changing anything: contacts are already fully owner-scoped for sales reps at every layer (list query, detail-page `notFound()` guard, the underlying API, and global search all already filter by the parent Account's `ownerUserId`) — no cross-rep visibility leak exists. What was actually missing was a way for admin to **act** on that: `PATCH /api/accounts/[id]` already supported admin-gated `ownerUserId` changes, but nothing in the UI called it. Added a bulk "Reassign owner to..." control next to the existing WhatsApp-sync bulk action on the Contacts list (admin-only, reuses the same checkbox-selection state), plus an Owner column so admin can see current ownership before reassigning. Resolves each selected contact's account (several contacts can share one) and PATCHes each unique account once, not once per contact.

Verified with 11 real-data + disposable-row assertions: stage velocity's per-stage `n` cross-checked against a raw `DealStageHistory` count, owner-scoped counts never exceeding the unscoped total, flooring classification against the actual label vocabulary, `getRepDeals()` against a real rep's roster, and a real account-reassignment write/readback — cleaned up after. Typecheck clean throughout.

## 2026-07-20 — CRM Analytics made admin-only, reversing Phase 5's self-scoped design

Explicit instruction after seeing a SALES login's All Tools panel: sales reps should not see analytics at all, only admin. This **reverses** a deliberate Phase 5 design choice (self-scoped CRM analytics so reps could see their own numbers) — not a bug fix, a product decision change, so it's logged rather than silently overwritten.

Sales had 4 separate ways to reach CRM analytics; all 4 gated, matching how `/team` (WhatsApp Sales Analytics) already works:
- `AllToolsPanel.tsx`'s "CRM analytics" tile — added the same `adminOnly: true` flag `/team` and `CRM settings` already use.
- `CrmTabs.tsx`'s "Analytics" tab (shown on every `/crm/*` page, plus `/leads` and `/deals`) — was rendered unconditionally with no role awareness at all. Now takes an `isAdmin` prop and filters the tab out; wired from `crm/layout.tsx` (added a `getCurrentUser()` call it didn't have before) and from `leads/page.tsx`/`deals/page.tsx`, which already had the role available.
- `crm/analytics/page.tsx` — was `requireUser()` (any logged-in role), so a sales rep typing the URL directly got straight through. Switched to `requireAdmin()`, the same helper `/users` already uses, instead of hand-rolling the check `/team/page.tsx` does inline.
- `GET /api/crm/analytics` — the real gap: previously never blocked, it self-scoped non-admin callers to `ownerIds: [user.id]` instead of rejecting them, so the API was reachable (with a rep's own data) even with every UI entry point hidden. Now returns `403 {error: "forbidden"}` for non-admin, same shape `/api/team/analytics` already returns. The now-dead self-scoping branch was removed rather than left unreachable.
- `crm/page.tsx`'s dashboard already had this right — its two analytics links only render inside the existing `isAdmin(user.role)` branch; the Sales "My Day" branch never had them. No change needed there.

Updated `scripts/rbac-matrix-check.ts`'s "CRM: analytics — self" row from a `PASS` (describing the old self-scoped behavior) to `NOT_IMPLEMENTED`, mirroring `/team`'s own "Analytics — self" row exactly — the row's `expected` column tracks the original build spec's ask (self-service analytics for every role), which hasn't changed; what changed is a second deliberate decision not to build it, same as `/team`. Re-ran the full matrix after: 8 pass, 0 fail, 8 not-implemented, 2 intentional divergences — up from 7/0/7/2, the one new not-implemented being this row switching from a stale PASS to an honest gap record.

No disposable-row DB script for this one — every change is either reusing an already-proven mechanism (`AllToolsPanel`'s existing `adminOnly` filter, `requireAdmin()`) or a single-line `isAdmin(role)` boolean check, with no data-shape logic to exercise. Verification here was a full grep sweep for every `crm/analytics` reference in the codebase (6 call sites found, all 6 accounted for above) plus a real run of `rbac-matrix-check.ts`, not a fabricated test.

## 2026-07-20 — Second post-launch feedback round

Five more issues from live use of the New Contact flow and analytics screens:

- **New Contact modal made genuinely fullscreen** (`AccountContactsClient.tsx`) — the previous `max-w-md` centered dialog was cutting off fields on real screens (shown in the user's screenshot). - **Bug fix: "Other" didn't reveal manual entry for every field, only some.** Customer Type's "Other" option wasn't wired to the same reveal-a-text-input pattern Designation already used — fixed by detecting the selected option's name case-insensitively (`=== "other"`) the same way, and reusing the pattern consistently rather than inventing a second one. The manual value is appended into the contact's `notes` as `"Customer type detail: ..."` rather than adding a dedicated column for what's an occasional free-text overflow, not a first-class field.

Ran 4 parallel investigations (ownership/assignment sync, `Conversation`↔`Deal` field mirroring, API-schema-vs-client-payload gaps, stale computed/cached values) against the actual code — every finding below was traced through real file:line evidence, not pattern-matched by name. 19 things surfaced total; fixed the 9 that were confirmed, live-triggered, and fixable without inventing a new product surface. The rest are logged in docs/DATA_GAPS.md with reasoning for why they weren't built now.

**Quotation delete never undid what quotation-create maintains.** The `isPrimary`-demotion fix (an earlier entry below) only covered creation. Deleting a quotation — including the deal's primary one — never re-promoted another revision or recomputed `Deal.quotedValue`, so a deal could end up with zero primary quotations (permanently undercounting "won" products, same failure mode as the original bug, just reached from DELETE) or a `quotedValue` pointing at a row that no longer exists. Added `reconcileDealAfterQuotationDelete()` (`src/lib/crm/deals.ts`) — checks the REMAINING rows' state rather than needing to know what was just deleted, so it's correct whether called after a single or bulk delete, and self-corrects even if called redundantly. Wired into both `DELETE /api/quotations/[id]` and the bulk `DELETE /api/quotations`.

**Conversation reassignment never touched the linked Deal, in either direction.** `PATCH /api/conversations/[id]` (an admin reassigning a customer to a different rep — a routine, common action) never propagated to `Deal.ownerUserId` at all; the deal just kept crediting the old rep forever, the exact same invisibility bug as the null-owner case, just reached by "reassign" instead of "never assigned." Fixed both directions: reassigning (or unassigning) a Conversation now syncs the linked Deal's owner, and — since there was previously no UI anywhere to reassign a Deal's owner directly either, despite the API already accepting it — added an admin-only Owner selector to the Deal Detail edit modal, which syncs back to the Conversation. Owner reassignment is admin-only on both sides now, matching the existing Conversation-reassignment rule exactly (a sales rep can edit everything else about their own deal, but handing it to someone else isn't theirs to decide).

**Chatbot-originated leads could never earn an owner.** `Lead` rows the chatbot creates correctly start with no owner (there's no real actor at automated-capture time — this part was correct, not a bug). But claiming one via the "Bot lead data sheet" only ever wrote `BotLead.assignedToUserId`; the mirrored general-CRM `Lead.ownerUserId` (linked via `BotLead.leadId`) was never touched, so every chatbot lead a rep claimed and worked stayed permanently invisible to `sourceAnalytics()`'s per-rep view in Team Performance. `PATCH /api/leads/[id]` now syncs the claim through to the linked `Lead` row.

**`PATCH /api/court-images/[id]` silently dropped fields its own `POST` sibling handles.** Found in code from earlier today: the wizard's `saveDraft()` sends `leadSourceId`/`customerProfileId`/`businessType` identically on create and update, `POST` declares and writes through all three, but the `PATCH` schema never declared them — Zod strips unknown keys silently, so re-saving an existing draft (the wizard's own "← Back" to Step 1 then re-save flow) looked successful but quietly lost any classification change. Same fields, same write-through logic, now on both schemas.

**Deal Detail edits never reached the Conversation they're paired with**, and a write-through precedence bug meant even the ONE existing sync path could regress a real quoted value back to a stale pre-quote estimate. `PATCH /api/deals/[id]` (the edit modal built earlier today) now writes expected-close-date and owner changes through to the linked Conversation. Separately, `transitionDeal.ts`'s own write-through had `estimatedValue` ahead of `quotedValue` in its fallback chain (`wonValue ?? estimatedValue ?? quotedValue ?? ...`) — backwards, since an actual quote is more concrete than a pre-quote guess; flipped the order. And the only way `Deal.quotedValue` ever reached `Conversation.dealValue` was piggybacked on `advanceDealStageIfEarlier`'s forward-only stage-advance — so revising a quote on a deal already past "Quotation Sent" (routine: a Negotiation-stage deal getting a re-priced revision) never re-synced it. Added a direct, unconditional sync in `POST /api/quotations` itself, independent of any stage logic.

**Deliberately not built** (logged in docs/DATA_GAPS.md rather than attempted under this pass's momentum): a UI for deleting a deal (schema-ready `deleted` flag, but delete is a meaningfully higher-risk feature than a sync fix and deserves its own consideration); an Account name/city edit screen (no `/api/accounts` route exists at all — a bigger feature, not a bug fix); `DealLineItem` snapshot resyncing on a quotation's line-item PATCH (confirmed dormant — no live caller sends `lineItems` via PATCH today, so it's a landmine for a future feature, not an active bug); `DealStageHistory.note` capture in the `/deals` closeout UI (cosmetic — history stays accurate, just less annotated); letting `findOrCreateDealForConversation`'s "reuse" branch silently reassign an *existing, legitimately-owned* deal to whoever touches it next (unlike the null-owner self-heal, this would be actively wrong — an admin creating a courtesy quote on behalf of a rep's deal must not steal it; explicit reassignment, just fixed above, is the correct mechanism for that, not implicit reuse).

Verified with 13 direct assertions against production/disposable test rows covering all 9 fixes, including that a non-primary quotation delete leaves an existing primary undisturbed (no false-positive re-promotion) and that the precedence fix actually changes the chosen value when both estimate and quote are present.

The forward map (`LEGACY_STAGE_MAP`) is many-to-one, so the reverse direction is inherently ambiguous — each old 7-stage slug resolves to the *earliest* (lowest sortOrder) new stage among its candidates, the most conservative read available without inventing detail the old system never captured. Never blocks or fails the legacy response — `/pipeline` must keep working exactly as it always has even if a sync attempt hits `transitionDeal()`'s own validation (e.g. "won" with no value on file yet); verified this exact case directly: the Deal still gets created, the stage correctly does NOT move, and the legacy Conversation-side response is unaffected.

**Dead code removed** — `src/app/api/conversations/[id]/deal/route.ts` had zero callers (confirmed by two independent searches); its entire field set was already a strict subset of what the `/stage` route persists in the same transaction. Deleted.

**Lead → Deal conversion — didn't exist at all.** No button, no API field, nothing ever wrote `Lead.convertedDealId`, which silently zeroed out `src/lib/analytics/sources.ts`'s lead-to-won tracing for every lead created after the one-time backfill. Added `POST /api/leads/[id]/convert` (distinct from the confusingly-named `PATCH /api/leads/[id]/route.ts`, which operates on the unrelated `BotLead` model) and a "Convert to deal" button on the General Leads tab. Chatbot-sourced leads are covered too, since they're already mirrored into a real `Lead` row on capture (see the Phase 1 entry below) — the General Leads tab shows all `Lead` rows regardless of origin.

All six fixes verified with 20 direct assertions against production (the Neon dev branch had expired — 1-day expiration set when it was created — so this pass tested safely against prod with `__TEST__`-prefixed rows, cleaned up immediately after, same pattern used throughout this build for anything the dev branch couldn't cover in time).

## 2026-07-15 — Phase 6: RBAC test matrix against spec §13 — findings, not silently fixed

Wrote `scripts/rbac-matrix-check.ts` to test every cell of the spec's §13 permission matrix against what's actually enforced. Result: `manager` and `management` were added to the role enum in Phase 0, but **no route anywhere in the app gives them any capability beyond what `sales` already has**, except the audit log built earlier today (`isManagerOrAbove`/`isManagementOrAbove` have exactly one real caller each: the audit-log route and its own declaration). Concretely, against every matrix row:

- **Others' leads/deals** (spec: MANAGER = CRUD own-office, MANAGEMENT = read-all): `GET`/`PATCH /api/deals[/[id]]` only branch on `isAdmin` — a MANAGER or MANAGEMENT user sees/edits exactly their own deals, same as SALES. No office-scoping exists at all (`Deal.officeId` is nullable and nothing populates it yet either).
- **Analytics — team/company** (spec: MANAGER = own office, MANAGEMENT = all): `/team` and its API are gated `role !== "admin"` — a MANAGER or MANAGEMENT user is blocked outright, same as SALES.
- **Reassign owner** / **Edit timeline markers** / **Change lead source post-creation** (spec: MANAGER+ only): none of these exist as distinct, gated actions. `ownerUserId` is just a regular field in `PATCH /api/deals/[id]`'s schema — the deal's OWNER (including a plain SALES rep) can already reassign it to anyone, which is *more* permissive than the matrix intends, not less. Timeline markers (`enquiryAt`, `siteVisitAt`, etc.) aren't editable via any endpoint today, by anyone.
- **Manage taxonomies** / **Manage users** / **View audit log**: correctly admin-only / management+ — these pass.
- **Manage rates/products** — a real conflict, not a gap: `PUT /api/quotations/rates`'s own header comment says *"Both admin and sales can edit per the product spec (‘editable by admin and sales also’)"* — a deliberate, pre-existing product decision, predating this CRM build. The new spec's matrix says rates should be ADMIN-only. **Not changed** — silently locking sales out of rate editing could break a workflow Fitoverse relies on (e.g. adjusting a rate live during a customer conversation) without their sign-off. Flagging for a decision rather than guessing which spec wins. `/api/products/*` (the actual Product catalogue, distinct from the rate sheet) IS already admin-only and matches the matrix correctly.

**Net effect: promoting a user to `manager` or `management` in `/users` today does almost nothing** — they get the same access as `sales`. Closing this properly (office-scoping, a real reassign-owner action, role-aware `/team` access) is a real feature build, not a hardening pass — flagged here rather than attempted piecemeal under Phase 6's time budget. See `docs/DATA_GAPS.md` #18.

## 2026-07-15 — Phase 5: WhatsApp bot commands

`User.phone` didn't exist — the spec's §10 assumed it ("identify the sender by `User.phone`") but no Phase 0-4 work added it. Added it nullable/unique, plus a self-service field on `/profile` (not an admin-set field on `/users` — a rep's bot-command number is inherently their own, and the admin `/api/users/[id]` route already blocks self-edits by design) so a rep opts in themselves rather than someone setting it for them.

**Confirm-before-write, one mechanism for both commands that need it.** `remind` and `stage` both require echoing back the parsed result before saving (spec explicit for `remind`; "with a confirm step" for `stage`). Added one small table, `PendingStaffAction` (one row per user, 10-minute expiry), rather than overloading the existing `ConversationFlow` model — that model's `path`/`currentStep` vocabulary is specific to the customer-facing product-enquiry flow, and force-fitting a "staff command" case into it would blur what that model means. A non-yes/no message while a confirmation is pending abandons it and processes the new command fresh, rather than leaving the user stuck (spec: "keep it small and forgiving").

**Command parsing is fully hand-rolled** (`src/lib/chatbot/staffParse.ts`), not a date-parsing library — no such dependency existed in the project, and the spec's own "keep it small and forgiving" framing fit a compact parser (today/tomorrow/weekday/"in N hours"/bare-time/ISO-fallback) better than pulling in a new package for one command. Verified with 30 direct unit-test cases, including the trailing-garbage-rejection edge case a naive substring-match implementation would get wrong ("tomorrow 9am call the client" must NOT parse as a valid date on its own — every date pattern is fully `^...$`-anchored, not just prefix-matched).

**`executeStaffCommand` never calls `sendText` directly** — the actual outbound send lives only in `handleStaffMessage`, which lazy-imports `@/lib/whatsapp` and the chatbot dispatcher at call time rather than at module load. This was originally about testability (the pure command-execution logic needed to be unit-testable against the dev database without the test script transitively pulling in `axios` and risking a real Meta API call), but it's a defensible structural split on its own merits, not just a workaround.

**Deferred, not built:** `new lead`'s source is unattributed (`leadSourceId: null`) — the command syntax has no source field and nothing else here indicates one. `quote <code>` links to the deal page rather than a quotation-builder deep link with a prefilled deal — no such query-param integration exists in the wizard today and building one was out of scope for a chat command that explicitly shouldn't attempt full quoting over chat anyway.

## 2026-07-15 — Phase 4: product analytics group by line label, not catalogue Product

`src/lib/analytics/products.ts` (spec §7.3) was built to group by the matched catalogue `Product.name`, falling back to "(unspecified)". Testing against real backfilled data showed every one of 163 line items landed in "(unspecified)" — checked the actual stored labels ("Fencing", "Sub Base", "Artificial Turf for Multisports", "Nylon Net") against real `Product.name` values ("Football Turf 50mm B-HO", "Professional Sports Lighting System") and confirmed these are two different vocabularies, not a matching bug. Quote line items are generated from the rate-sheet cost categories (turf/fencing/lighting/sub-base/nets, always present, computed by sqft × rate); `productId` is a separate, optional field set only when a rep explicitly picks a specific catalogue SKU to illustrate one line — most quotes never do this. This isn't a historical-data-only gap; it's how the wizard works going forward too.

Fixed: group by `product?.name ?? label ?? "(unspecified)"` instead of requiring a Product match. `label` is the rate-sheet category name and is consistent across quotes, so grouping by it already answers the spec's own headline example ("football turf enquiries were higher and we also closed 4 football turf projects") without needing every line tied to a catalogue SKU. Verified: 24 real monthly rows across 12 categories with real rupee totals, vs. one meaningless "(unspecified)" bucket before the fix.

## 2026-07-15 — Phase 4: historical backfill (`scripts/backfill-crm.ts`)

Phase 1-2 shipped `Deal`/`Account`/`Lead` as new, empty tables — every quotation/conversation/design/bot-lead created before this build stayed unlinked, so the first Phase 4 analytics screens (sales-activity, funnel, geography, customers) read as entirely empty even though ~30 real quotations exist. Built the backfill script the original plan already called for but hadn't been run yet.

**Grouping rule, extended from the plan.** The plan said "one Deal per Conversation with quotes/designs" — but only 1 of 30 quotations actually has a `conversationId` (most quotes in this tool are created standalone, not threaded through a WhatsApp conversation). Extended the rule: conversation-grouped where a `conversationId` exists (all of that conversation's quotes/designs share one Deal), plus one Deal per conversation-less quotation/design otherwise. Idempotent either way — re-running looks up `dealId`/`leadId == null` and reuses an existing Deal for a conversation rather than duplicating it.

**Account de-dup: exact case-insensitive name match**, same rule the live "possible duplicate" check in `POST /api/deals` uses. Verified against real data: 18 backfilled deals correctly collapsed onto 12 accounts (e.g. "wr"/"WR" quotes merged onto one account). Known limitation: short/generic walk-in names ("e", "gh", "sf") could theoretically merge two different real customers who happen to share a name — accepted as a best-effort reconstruction, not verified lineage; fixable later by re-parenting a Deal to a different Account.

**Stage assignment from known signal only.** A deal lands in "Quotation Sent" if any of its linked quotations has `status: "sent"`, else "Enquiry Received" (the earliest stage). `outcome`/`closedAt` are left null on every backfilled deal — whether a historical quote became a won or lost project isn't recoverable from existing data and was not invented.

**Line items best-effort name-matched to `Product`.** Historical `Quotation.lineItems` JSON predates the `productId` field (dropped by the bug fixed in Phase 2), so line names are matched case-insensitively against `Product.name` where possible; unmatched items keep `productId: null` rather than guessing.

Tested on the `crm-dev` Neon branch first (30/30 quotations, 4/4 designs, 7/7 bot-leads linked; sales-activity and funnel screens confirmed showing real numbers, including 6 sent-revisions against 1 unique deal on the one conversation-linked record — the exact "revisions vs. unique deals" distinction the spec asked for). Then run against production — additive only (fills null FKs, creates new Account/Deal/Lead/DealLineItem rows), touches no existing row's existing data, safe to re-run.

## 2026-07-15 — Phase 0 reconciliation decisions

**Naming: `Contact` → `AccountContact`.** The spec's "person at an account" model collides with an existing, unrelated, actively-used `Contact` model (the broadcast/marketing contact list — phone, consent flag, tags). The new model is `AccountContact`.

**Migration tooling: baseline Prisma Migrate before any schema change.** This project has only ever used `prisma db push` — no `prisma/migrations/` history exists. Before any CRM schema work, migration history gets baselined (`prisma migrate diff --from-empty ... | prisma migrate resolve --applied`) so every subsequent change is a reviewable, named migration. All schema changes are tested against a Neon branch (not production directly) first.

**Money: `Decimal(12,2)` rupees, not paise integers.** Every existing money column in this schema is Decimal-rupees. New fields (`Deal.estimatedValue`/`quotedValue`/`wonValue`, `DealLineItem.rate`/`amount`, `Product.currentRate`) follow the same convention rather than introducing a second unit system. Same for GST: `Product.gstPercent Decimal(5,2)`, not `gstRateBp` basis points — matches the existing `RateSheetItem.gstPercent`/`QuoteLineItem.gstPercent` convention.

**User-reference FKs use the existing `...UserId` suffix**, not the spec's `...Id` — 8 existing precedents (`assignedToUserId`, `changedByUserId`, `ownerUserId`, etc.). `DealStageHistory.changedByUserId` is a structural match for the existing `PipelineStageHistory.changedByUserId`.

**Roles: widen, don't replace.** `User.role` stays a plain String; the Zod enum and shared `Role` type (`src/lib/rbac.ts`) widen from `"admin"|"sales"` to include `"manager"|"management"`. Public self-signup (`/api/auth/signup`) deliberately keeps its enum at 2 values — manager/management are granted only via the admin-controlled `/users` page.

**Pipeline → Deal: write-through, not replacement.** `Deal` becomes the new source of truth (Phase 1), but `transitionDeal()` writes through to `Conversation.pipelineStage`/`dealValue`/etc. whenever a Deal has a `conversationId`, so `/pipeline` and the current `/team` analytics keep working unchanged through Phase 1-2. The formal cutover of those pages to read `Deal` directly happens in Phase 4. `PipelineStageHistory` is not renamed/dropped — it stays serving the legacy view; new history goes to `DealStageHistory`.

**GST: `gstMode` column added, `"FLAT_18"` compute path deferred.** `Quotation.gstMode` defaults to `"SPLIT"` (today's itemized-bracket behavior — zero change). What a flat-18% quote should actually mean operationally is a real business/tax question, not something to guess at — left unimplemented until Fitoverse confirms.

**Taxonomy colors render via inline `style`/CSS custom properties**, never dynamically-built Tailwind class names — avoids the trap the existing `STAGE_COLOR_CLASSES` 7-token lockdown exists to work around (Tailwind's JIT purger can't see dynamic class names).

**`Quotation.lineItems` (JSON) and `DealLineItem` (relational) coexist.** The JSON snapshot stays the PDF-rendering source of truth; `DealLineItem` is a new, parallel, denormalized analytics projection with real `productId`/`sportId` FKs.

**Nullability: every new FK/column stays nullable, indefinitely.** Matches the codebase's own established pattern (visible on `Conversation`'s pipeline fields) — additive only, no later "tighten to NOT NULL" pass planned.

See `docs/DATA_GAPS.md` for the business inputs still needed from Fitoverse before certain features (value bands, city tiers, stage probabilities, etc.) are fully meaningful.
